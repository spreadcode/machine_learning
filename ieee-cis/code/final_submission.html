Stop worrying about DeviceType and DeviceInfo!¶

But a quick look at 2 things convinced me that special handling of these 2 columns is not warranted. Why? Large number of nans (80%+) tell us that this is not conclusive Given that MacOS and iOS devices are usually far more expensive than Windows desktops, it is counter-intuitive that a fraudster will use an expensive machine to steal unless iOS and MacOS devices make it easier to commit fraud than windows devices. From my knowledge of security, this is tenuous. But Item 1 was the dominant reason. It also informed my overall strategy of: -Dropping useless columns -Implementing a deep learning NN

Find and remove useless columns¶

If many (defined below) column values are the same, remove the column If many (defined below) column values are null, remove the column

In [44]:
null_val_threshold = 0.9
repeated_val_threshold = 0.9

def get_too_many_null_attr(data):
    many_null_cols = [col for col in data.columns if data[col].isnull().sum() / data.shape[0] > null_val_threshold]
    return many_null_cols

def get_too_many_repeated_val(data):
    big_top_value_cols = [col for col in train_df.columns if train_df[col].value_counts(dropna=False, normalize=True).values[0] > repeated_val_threshold]
    return big_top_value_cols

def get_useless_columns(data):
    too_many_null = get_too_many_null_attr(data)
    print("More than {} nulls: ".format(null_val_threshold) + str(len(too_many_null)))
    too_many_repeated = get_too_many_repeated_val(data)
    print("More than {} repeated values: ".format(repeated_val_threshold) + str(len(too_many_repeated)))
    cols_to_drop = list(set(too_many_null + too_many_repeated))
    cols_to_drop.remove('isFraud')
    return cols_to_drop

cols_to_drop = get_useless_columns(train_df)
train_df = train_df.drop(cols_to_drop, axis=1)
test_df = test_df.drop(cols_to_drop, axis=1)

    for x in list(in_df.columns.values):

            median = float(in_df[x].median())
            std =  float(in_df[x].std())
            in_df[x] = np.where( (in_df[x] - median) > 4*std, median, in_df[x])

Numerical Scaling¶

Scale the data so that it will be in a more NN friendly format. First we will do log1p for any values that are above 100 and not below 0. This is in order to scale down any numerical variables that might have some extremely high values that hurts the statistics of the standard scaler

https://www.kaggle.com/kernels/scriptcontent/18546295/download#Keras-NN-Starter-Kernel

https://www.kaggle.com/vaishvik25/keras-nn

The idea came from other kernels above: the use of standard scaler makes sense so that the values have a normal mean and std. This makes the NN converge signficantly faster + prevents crashes/freezes/hangs.

We will then pass the embeddings through a spatial dropout layer which will drop dimensions within the embedding across batches and then flatten and concatenate. Then we will concatenate this to the numerical features and then add some more dense layers after.

Create model- basic model with special handling of categoricals¶

Use Dropout to prevent overfitting use an embedding layer; vectorize all inputs -> flatten them use log1p to better support small input values

Neural Network Model Details

Our neural network will: use an embedding layer for categoricals numericals will go through feed forward dense layers. We create our embedding layers such that we have as many rows as we had categories and the dimension of the embedding is the log1p + 1 of the number of categories. So this means that categorical variables with very high cardinality will have more dimensions but not signficantly more so the information will still be compressed down to only about 13 dimensions and the smaller number of categories will be only 2-3.

We will then pass the embeddings through a spatial dropout layer which will drop dimensions within the embedding across batches and then flatten and concatenate. Then we will concatenate this to the numerical features and add some more dense layers after.

Activation functions¶

I used relu for efficiency with sigmoid in the end to ensure that the output layer doesn't blow up+to mitigate the "vanishing" gradient issue with relu. Note: I tried softmax for the output layer as advised by the reviewer of my project proposal. Unfortunately, that yielded AOC of 0.5 so I dropped it in favor of sigmoid. Thank you for the input anyway.

def make_model():
    k.clear_session()
    categorical_inputs = []
    for cat in categorical:
        categorical_inputs.append(Input(shape=[1], name=cat))

    categorical_embeddings = []
    for i, cat in enumerate(categorical):
        categorical_embeddings.append(
            Embedding(category_counts[cat], int(np.log1p(category_counts[cat]) + 1), name = cat + "_embed")(categorical_inputs[i]))

    categorical_logits = Concatenate(name = "categorical_conc")([Flatten()(SpatialDropout1D(.1)(cat_emb)) for cat_emb in categorical_embeddings])

    numerical_inputs = Input(shape=[tr_df[numerical].shape[1]], name = 'numerical')
    numerical_logits = Dropout(.1)(numerical_inputs)


    x = Concatenate()([
        categorical_logits, 
        numerical_logits,
    ])
    
    x = Dense(200, activation = 'relu')(x)
    x = Dropout(.2)(x)
    x = Dense(100, activation = 'relu')(x)
    x = Dropout(.2)(x)
    out = Dense(1, activation = 'sigmoid')(x)


    model = Model(inputs=categorical_inputs + [numerical_inputs],outputs=out)
    loss = "binary_crossentropy"
    model.compile(optimizer=Adam(lr = 0.005, amsgrad=False), loss = loss)
    return model
